## 一、产品目标

通过引入 AI Agent 混合模式（阶段1：正则扫描 + 阶段2：AI 深度分析兜底），解决当前 AI 代码分析功能在**非标准路由定义场景下无法匹配**的问题，同时提升用户体验（实时日志输出、进度可视化）和分析深度（完整调用链路分析）。

---

## 二、用户故事

1. **作为测试工程师**，我想要 AI 不仅能找到接口对应的 Handler 函数，还能分析出 Handler 调用了哪些服务、哪些数据库操作，以便于生成更完整的接口自动化测试用例（包括 mock 数据和断言）。

2. **作为测试工程师**，我想要点击"分析"后能看到实时的分析进度和日志输出（类似 CI/CD 构建日志），以便于了解分析过程是否正常、预估剩余时间、及时发现异常。

3. **作为开发工程师**，我想要 AI 分析的结果包含带参数的 curl 命令和断言示例，以便于快速验证接口是否符合预期。

4. **作为产品经理**，我想要评估是否需要为用户提供自定义编辑 prompt 的能力，以便于在通用分析和个性化需求之间找到平衡。

5. **作为系统管理员**，我想要确保新功能的代码改动不会影响现有功能的稳定性，以便于平滑升级和系统维护。

## 三、需求池

### P0（必须有）

#### 需求 1：阶段1 快速扫描（保留现有实现）
**描述**：保留当前的正则扫描逻辑，用于快速匹配标准路由定义。

**验收标准**：
- [ ] 阶段1 扫描逻辑与现有实现保持一致（正则匹配 `RegisterRoutes`、`.GET(` 等）
- [ ] 扫描结果包含：文件路径、Handler 函数名、路由路径
- [ ] 扫描进度通过 IPC 实时推送到渲染进程
- [ ] 扫描时间在 1-2 秒内完成（中型仓库，500 个文件）

#### 需求 2：阶段2 AI 深度分析（新增）
**描述**：对阶段1 的匹配结果，使用 AI Agent 模式进行深度分析，生成完整的调用链路报告。

**验收标准**：
- [ ] AI 分析包含：Handler 函数、调用的 Service/Model、数据库操作、外部服务调用
- [ ] 分析报告生成**3 个场景**的调用链路（正常流程、参数校验失败、权限校验失败）
- [ ] 分析报告生成带参数的 curl 命令示例（多行，带换行符 `\`）
- [ ] 分析报告生成 Python 断言示例（使用 `requests` 库，包含 `assert` 语句，有注释）
- [ ] 分析报告格式为 JSON，包含场景链路表格数据、curl 命令、Python 断言
- [ ] 分析时间在 3-8 秒内完成（1-2 次 AI API 调用）

#### 需求 3：阶段1 失败兜底（新增）
**描述**：如果阶段1 未找到匹配，允许用户选择"深度搜索"，触发 AI Agent 完整推理链。

**验收标准**：
- [ ] 阶段1 失败时，前端显示"未找到匹配，是否启用深度搜索？"提示
- [ ] 用户确认后，触发 AI Agent 完整推理链（工具调用循环）
- [ ] 兜底分析时间在 5-15 秒内完成（最多 10 次工具调用）
- [ ] 兜底分析成功後，展示与阶段2 相同格式的分析报告

#### 需求 4：分析进度页面（新增）
**描述**：点击"分析"后跳转到分析进度页面，实时显示分析进度和日志输出。

**验收标准**：
- [ ] 点击"分析"按钮后，跳转到分析进度页面
- [ ] 页面实时显示分析进度（百分比或阶段描述）
- [ ] 页面实时显示分析日志（类似 CI/CD 构建日志，自动滚动）
- [ ] 分析完成后，页面展示分析结果（或跳转到结果页面）
- [ ] 分析失败时，页面显示错误信息和重试按钮

**右侧：Python 断言结果**
- 使用 `requests` 库
- 包含 `assert` 语句
- 有注释说明
- 固定高度 **200px**
- 超出部分显示**垂直滚动条**

**代码示例**：

左侧（curl）：
```bash
curl -X GET \
  "http://localhost:8080/api/v1/users/1" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer ${TOKEN}"
```

右侧（Python 断言）：
```python
import requests

response = requests.get(
    "http://localhost:8080/api/v1/users/1",
    headers={"Authorization": "Bearer ${TOKEN}"}
)

# 断言状态码
assert response.status_code == 200

# 断言响应字段
data = response.json()
assert "id" in data
assert data["id"] == 1
```

### 5.2 产品功能相关

5. **是否需要自定义编辑 Prompt 的功能？**
   - 选项 A：需要（满足高级用户的个性化需求）
   - 选项 B：不需要（保持产品简洁，降低维护成本）
   - **分析**：
     - 支持自定义 Prompt 的优点：灵活、可扩展、满足高级用户需求
     - 支持自定义 Prompt 的缺点：增加产品复杂度、需要 Prompt 版本管理、可能导致分析结果不稳定（用户编辑错误）
   - **建议**：**暂时不需要**（P2 需求）。原因：
     1. 当前阶段目标是验证混合模式的可行性，自定义 Prompt 不是核心需求
     2. 如果用户有个性化需求，可以通过"反馈建议"渠道收集，后续迭代考虑
     3. 自定义 Prompt 需要配套的错误处理、版本管理、默认值重置等功能，增加开发成本

6. **分析报告是否需要支持导出？**
   - 选项 A：需要（导出为 Markdown、PDF、HTML）
   - 选项 B：不需要（只支持在线查看和复制）
   - **建议**：**需要（P1 需求）**。原因：
     1. 用户可能需要将分析报告集成到测试文档中
     2. 导出为 Markdown 可以实现为"一键复制"，成本较低

7. **是否需要支持多仓库分析？**
   - 选项 A：需要（一次分析多个仓库的调用链路）
   - 选项 B：不需要（只支持单仓库分析）
   - **已确认**：选项 B（不需要）✅。理由：当前需求是"精准分析接口对应的代码及链路逻辑"，单仓库已经满足。

     2. 如果用户需要查看某条日志，可以通过"滚动回看"实现

10. **分析结果页面是否需要支持分享？**
    - 选项 A：需要（生成分享链接，发送给团队成员）
    - 选项 B：不需要（只支持本地查看）
    - **建议**：**不需要**（P2 需求）。原因：
      1. 当前应用是桌面应用，不支持云端同步
      2. 分享功能需要配套的用户系统、权限管理等功能，复杂度较高

### 5.4 兼容性和迁移相关

11. **新功能是否需要兼容旧版本的分析结果？**
    - 选项 A：需要（旧版本的分析结果可以在新版本中查看）
    - 选项 B：不需要（新版本发布后，旧分析结果自动失效）
    - **建议**：**需要**（P1 需求）。原因：
      1. 用户可能已经保存了一些分析结果，如果新版本不兼容，会导致数据丢失
      2. 兼容性实现方式：分析结果保存为 JSON 文件，新版本读取时做版本号判断

12. **新功能是否需要在原有代码基础上重构？**
    - 选项 A：需要（重构现有代码，使其更易于扩展）
    - 选项 B：不需要（在原有代码基础上增量修改）
    - **建议**：**部分重构**（P0 需求）。原因：
      1. 当前实现中，AI 分析逻辑集中在 `ipc.ts` 中，需要抽离为独立的服务（如 `AIAnalyzeService`）
      2. 重构范围：IPC 通道、AI 分析服务、前端状态管理

### 5.5 成本和性能相关

13. **是否需要限制 AI API 调用次数？**
    - 选项 A：需要（防止用户滥用，导致成本过高）
    - 选项 B：不需要（用户使用自己的 API Key，成本由用户承担）
    - **建议**：**需要**（P1 需求）。原因：
      1. 即使用户使用自己的 API Key，也应该提供"预估成本"提示，避免意外高额账单
      2. 限制方式：每次分析前，弹窗显示"预估成本"，用户确认后继续

14. **是否需要缓存分析结果？**
    - 选项 A：需要（相同请求路径和仓库版本，直接返回缓存结果）
    - 选项 B：不需要（每次分析都重新执行）
    - **建议**：**需要**（P1 需求）。原因：
      1. 缓存可以减少 AI API 调用成本
      2. 缓存策略：基于请求路径 + 仓库 commit hash 生成缓存 key
